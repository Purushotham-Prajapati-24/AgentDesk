## Prerequisites

Before you start, you'll need:

- **Documents to ingest** — help articles, policy PDFs, product manuals, FAQs. The chatbot is only as good as this corpus.
- **A vector database** — Qdrant (self-hostable), Pinecone (managed), Weaviate, or pgvector if you're on Postgres.
- **An embedding model** — OpenAI's `text-embedding-3-small`, or an open-source alternative like `all-MiniLM-L6-v2`.
- **A generation LLM** — any model that follows system prompts (GPT-4o, Claude, Llama, etc.).
- **A language runtime** — Python (most common for RAG) or TypeScript/Node.js.

## Step 1 — Choose an embedding model

The embedding model converts text into a fixed-length vector. This vector is what the database searches against. The critical rule: **use the same model for indexing and querying.** Mixing models produces garbage results.

| Model | Dimensions | Hosted | Best for |
| --- | --- | --- | --- |
| `text-embedding-3-small` | 1536 | OpenAI API | Default choice, best quality-to-cost |
| `text-embedding-3-large` | 3072 | OpenAI API | Higher accuracy, higher cost |
| `all-MiniLM-L6-v2` | 384 | Self-hosted | Privacy-sensitive, low-resource |
| `bge-large-en-v1.5` | 1024 | Self-hosted | Strong open-source quality |

```python
from openai import OpenAI
client = OpenAI()

def embed(text: str) -> list[float]:
    response = client.embeddings.create(
        input=text,
        model="text-embedding-3-small"
    )
    return response.data[0].embedding
```

## Step 2 — Set up a vector database

Create a collection whose vector size matches your embedding model's output dimensions.

```python
from qdrant_client import QdrantClient
from qdrant_client.models import Distance, VectorParams

client = QdrantClient(url="http://localhost:6333")

client.create_collection(
    collection_name="support_docs",
    vectors_config=VectorParams(size=1536, distance=Distance.COSINE)
)
```

**Tenant isolation tip:** store a `tenant_id` payload field on every point and filter by it at query time. This prevents cross-tenant retrieval — a hard requirement for B2B support. AgentDesk does this with a separate Qdrant collection per tenant.

## Step 3 — Chunk your documents

Chunking is the most underrated step. Bad chunking produces bad retrieval regardless of model quality.

Use **recursive text splitting**: break by headers first, then paragraphs, then sentences, targeting 500–1000 tokens per chunk with 50–100 tokens of overlap. Preserve tables and code as atomic units — never split a table mid-row.

```python
from langchain.text_splitter import RecursiveCharacterTextSplitter

splitter = RecursiveCharacterTextSplitter(
    chunk_size=800,
    chunk_overlap=100,
    separators=["\n\n", "\n", ". ", " ", ""]
)

chunks = splitter.split_text(document_text)
```

## Step 4 — Embed and store the chunks

Run each chunk through the embedding model and upsert into the vector database with metadata.

```python
points = []
for i, chunk in enumerate(chunks):
    vector = embed(chunk)
    points.append({
        "id": f"doc_{doc_id}_chunk_{i}",
        "vector": vector,
        "payload": {
            "text": chunk,
            "source": doc_filename,
            "page": page_number,
            "tenant_id": tenant_id,
        }
    })

client.upsert(collection_name="support_docs", points=points)
```

## Step 5 — Build the retrieval prompt

On each query: embed the question, search for the top-k similar chunks (filtered by tenant), and construct a system prompt that grounds the answer in the retrieved context.

```python
def retrieve(query: str, tenant_id: str, k: int = 5):
    query_vector = embed(query)
    results = client.search(
        collection_name="support_docs",
        query_vector=query_vector,
        query_filter={"must": [{"key": "tenant_id", "match": {"value": tenant_id}}]},
        limit=k
    )
    return [hit.payload["text"] for hit in results]

def build_prompt(query: str, context_chunks: list[str]) -> str:
    context = "\n\n".join(context_chunks)
    return f"""Answer the user's question using only the following context.
If the context doesn't contain the answer, say "I don't know" and suggest escalating to a human.

Context:
{context}

Question: {query}"""
```

## Step 6 — Wire up the chat loop

Connect the retrieval + generation pipeline to a chat interface. Store conversation history for multi-turn context.

```python
def chat(query: str, tenant_id: str):
    chunks = retrieve(query, tenant_id)
    prompt = build_prompt(query, chunks)

    response = client.chat.completions.create(
        model="gpt-4o",
        messages=[
            {"role": "system", "content": prompt},
            {"role": "user", "content": query}
        ]
    )
    return response.choices[0].message.content
```

## Production considerations

The MVP above works. Production requires more:

- **Hybrid search.** Combine vector similarity with keyword (BM25) search. Pure-vector retrieval misses exact-match queries (order numbers, SKUs). Qdrant and Weaviate support this natively.
- **Reranking.** After retrieving top-20 chunks, rerank with a cross-encoder (e.g., Cohere Rerank) and keep the top-5. Improves precision materially.
- **Observability.** Log every query, retrieved chunks, similarity scores, and the generated answer. You can't fix what you can't see.
- **Source citation.** Return the source document and chunk with every answer. Customers and operators need to verify.
- **Rate limiting and abuse prevention.** Public chatbots get spammed. Add per-IP and per-session limits.
- **Tenant isolation.** If you serve multiple customers, enforce strict per-tenant retrieval filtering. A leak here is a data breach.

## Common pitfalls

- **Garbage in, garbage out.** The #1 cause of bad RAG is bad source documents. If your help articles are outdated or contradictory, the chatbot will surface that. Invest in your knowledge base first.
- **Chunking too small or too large.** Chunks under 200 tokens lose context; chunks over 2000 tokens dilute relevance. 500–1000 is the sweet spot for most content.
- **No fallback.** A RAG chatbot without a confidence threshold *will* hallucinate on out-of-scope questions. Always have an escalation path.
- **Ignoring hybrid search.** Pure-vector retrieval fails on exact identifiers. Add keyword search.
- **Not measuring.** If you aren't logging retrieval scores and tracking escalation rates, you're flying blind.
